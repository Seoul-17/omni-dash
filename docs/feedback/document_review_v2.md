**기획서 약속**: "참고 데이터 셋: 5개 대표 입력 형태 데이터" (P0).
**해커톤 평가 영향**: 범용성 25점 + 자동 생성 25점 = 50점에 직접 영향. 심사위원이 "다양한 데이터 형태 대응"을 검증할 수단이 없음.

**보완안**: `docs/skills/examples/` 신설.
```

├── 01_kr_equity.csv          # 국내 주식 단일 (005930.KS, KODEX 200 등)

├── 05_messy_input.csv        # 결측치·이상치 포함 (정규화 강제 검증)
└── outputs/
    ├── 01_kr_equity.output.json
    ├── 02_global_etf.output.json
    └── ...
```

## 7. v1.1 → v2 회고

### 잘된 점
- v1.1 보완안 §3.1~§3.5 직접 명세 초안이 그대로 이식 가능한 수준이어서 작업 마찰 최소화.
- Quick/Standard/Full + §6.2 모드별 필드 채움 표 조합으로 "출력 모드"가 단순 UX 개선이 아니라 출력 계약과 연동된 기능임을 입증.
- §9.1에 Quick·Standard 두 입력 사례를 모두 적은 점은 §0.5 모드 자동 결정 로직을 검증하는 효과적 장치.

### 보완 필요
- **단일 파일 정책**과 **400라인 룰**이 충돌. v2에서 references/ 분리로 해결 권장.
- E2E 예시가 1.5세트(Quick 입력만 + Standard 풀세트)로 절반만 충족. Quick 출력 JSON이 명세 검증의 마지막 퍼즐.
