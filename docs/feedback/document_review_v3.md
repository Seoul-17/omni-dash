## 2. v2가 놓친 새로운 발견 (v3 New Findings)

### 2.1 [P0] 출력 계약(§6.1) 타입 시스템 내재적 결함

**A. 필드 중복 — `tracking_error`**
```typescript
interface BenchmarkComparison {
  tracking_error: number;          // ← 여기
  ...
}
interface SemanticMetrics {
  benchmark: { id: string; performance: number; tracking_error: number };  // ← 여기에도
}
```
같은 값이 두 경로로 전달 가능 → 어느 쪽이 source of truth인지 불명. 프론트가 어느 필드를 읽어야 할지 임의 결정.

**B. 필드 중복 — `sortino`**
```typescript
interface RiskMetrics { sortino: number; ... }       // §6.1 RiskMetrics
interface SemanticMetrics {
  risk_adjusted: { sortino: number; ... };           // §6.1 SemanticMetrics.risk_adjusted
}
```
동일 지표가 두 객체에 동시 존재. §2.1 "Single Source of Truth" 원칙을 출력 계약이 자체 위반.

**C. enum 명명 불일치**
- `Insight.severity: "info" | "warn" | "critical"`
- `RiskMetrics.asset_class_thresholds: "ok" | "warn" | "critical"`

`info` ↔ `ok` 의미 동일하나 명명 분리. 프론트 색상 매핑 시 두 enum을 별도 처리해야 함.

**보완안 (skills_v2.md용)**:
- `tracking_error`는 `BenchmarkComparison`에만 두고 `SemanticMetrics.benchmark`에서 제거
- `sortino`는 `RiskMetrics`에만 두고 `SemanticMetrics.risk_adjusted`에서 제거 (또는 그 반대 — 한쪽으로 통일)
- enum 통일: `"info" | "warn" | "critical"`로 표준화하고 `RiskMetrics.asset_class_thresholds`도 동일 사용

### 2.2 [P0] 시맨틱 레이어(§2) 누락 표준 지표

기획서가 "검증된 분석 규칙"을 약속했고 개요.md 평가기준 "Skills.md 설계 25점"이 분석 규칙 명확성을 직접 채점하는데, 다음 표준 지표가 부재:

| 누락 지표 | 중요도 | 사유 |
|---|---|---|
| **VaR (Value at Risk, 95%)** | 높음 | §6.1 `RiskMetrics`에 `var_95` 필드는 있으나 §2 시맨틱 레이어에 정의 없음 — 환각 방지 가드레일 위반 |
| **CVaR / Expected Shortfall** | 높음 | 꼬리 위험(tail risk) 측정 표준. §5.2 트리거에 "꼬리 위험"을 언급하면서 정의 지표가 없음 |
| **칼마 비율 (Calmar Ratio)** | 중간 | `CAGR / |MDD|`. 헷지펀드·자산배분 평가 표준 |
| **회복 기간 (Recovery Period)** | 중간 | MDD를 측정하면서 회복 기간을 측정하지 않음 — 절반의 명세 |
| **롤링 윈도우 지표** | 중간 | 30일/90일 롤링 샤프 등. §3.1 차트 매트릭스에 "롤링" 옵션 없음 |
| **상관계수 안정성** | 낮음 | 시점 간 상관관계 변동 측정 |

**보완안**: §2.3에 위 4개 지표 추가 (각 1행씩, 약 30라인 증가). VaR는 이미 출력에 있으므로 정의 누락은 즉각 수정 필요.

### 2.3 [P0] 거래내역 vs 잔고(Holdings) 데이터 구분 부재

**현황**: §1.1 `portfolio` 스키마는 잔고(현재 보유 상태)만 정의:
```yaml
portfolio:
  position_id, asset_id, quantity, avg_cost, weight, acquired_at
```

**갭**: 거래내역(transaction history)과 잔고는 다른 데이터:
- 잔고: 현재 시점 스냅샷 (위 스키마)
- 거래내역: 매수/매도 이벤트 시계열 (`buy_sell, price, quantity, transaction_date, fees`)

기획서 §1 "MTS, 엑셀, 노트 등에 기록이 파편화"는 **거래내역**을 시사함. 사용자가 매수/매도 기록 CSV를 올리면 시스템이 잔고로 변환해야 하는데, 그 변환 명세가 없음.

**보완안**: §1.1에 `transactions` 스키마 추가 + §1.2에 "거래내역 → 잔고 변환 규칙" 추가.
```yaml
transactions:
  transaction_id: string
  asset_id: string
  side: enum [buy, sell, dividend, split]
  quantity: number
  price: number
  fees: number
  executed_at: string  # ISO-8601
```

### 2.5 [P1] 시연 시나리오 작동에 필요한 명세 부재 (기획서 §5 정합성)

기획서가 시연으로 약속한 3가지가 명세상 작동 불가:

**시연 1 — "양식이 엉망인 CSV 업로드 → 3초 만에 대시보드"**
- 갭: §0.5에 Standard 모드 응답 시간 목표는 **5초**. 시연 약속 3초와 충돌.
- 갭: "양식이 엉망인" 입력의 자동 컬럼 매핑 (예: `종목코드` → `asset_id`) 명세 부재. §1.1은 표준 스키마만 정의.

**시연 2 — "전혀 다른 스키마 추가 업로드 → 동일 레이아웃"**
- 갭: 다중 포트폴리오 **병합 정책** 명세 부재. 두 포트폴리오를 합산할지 별도 비교할지, 통화 다를 때 어떻게 처리할지.

**시연 3 — "Skills.md 텍스트 수정 → 즉시 반영"**
- 갭: **핫리로드 / 캐시 무효화** 명세 부재. version 변경 시 기존 캐시를 어떻게 무효화하는지.
- 갭: 어느 필드 수정이 어느 출력에 영향을 주는지의 의존성 그래프 부재.

**보완안**: §11(신설) "시연 보장 명세" 또는 §1.4 "입력 어댑터" + §6.3 "캐시 정책" 추가.

### 2.6 [P2] 응답 시간 목표(§0.5)의 현실성

**현황**: Quick < 1.5s / Standard < 5s / Full < 15s

**문제**:
- LLM API 호출만 통상 2~5s (특히 한글 응답 생성 시 토큰 길이↑)
- §6.1 출력 계약 전체를 LLM이 생성한다면 Full 15s도 빠듯
- 시연 1의 3초 목표(기획서)와도 명세상 일치하지 않음

**보완안 옵션**:
- A. 시간 목표를 "p50 / p95" 분리 명시 (예: Quick p50<2s, p95<5s)
- B. "캐시 적중 시" / "콜드 스타트" 분리
- C. 비현실적 시간 목표 제거하고 "사용자 체감 단계" 정성 묘사

### 2.7 [P2] 타겟 사용자(초보 투자자) ↔ 명세 톤 미스매치

**기획서 §1**: 타겟 = "초보 투자자(개미)"
**skills_v1.md §2.3**: 트레이너 비율, 정보 비율, 알파, R² 등 **고급 지표 11개** 나열

**갭**: 초보 투자자는 트레이너 비율이 무엇인지조차 모름. §5.4에 "C-레벨/애널리스트/일반 투자자" 분기는 있으나 출력 모드(Quick/Standard/Full)와 직교적이며, "일반 투자자" 케이스에서 어떤 지표가 보여지고 어떤 용어로 변환되는지 명세 부재.

**보완안**:

### 2.8 [P3] 자동 검증·테스트 케이스 명세 부재

**현황**: §7.1·§7.2가 "금지/필수" 항목 나열. 그러나 어떻게 자동 검증하는지 부재.

**보완안**: §7.4 "검증 케이스" 신설 — 입력 X에 대해 출력 Y가 나와야 한다는 골든 케이스 5개 정의. examples/ 디렉토리(v2 §3.1)와 결합.
