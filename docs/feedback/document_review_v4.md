### 2.2 [P0] §2.6 audience 자동 결정 로직 차원 오류

**현황 (§2.6)**: "단순 입력 → novice / 복잡 입력 → intermediate"
**문제**: `audience`는 **사용자의 금융 지식 수준**의 함수이지, **데이터 복잡도**의 함수가 아님. 단순한 데이터를 올린 expert도 있고, 복잡한 데이터를 올린 novice도 있음. 차원이 잘못됨.

**증거**: §1.1 `analysis_config.audience`가 사용자 입력 필드로 정의되어 있는데, §2.6에서 자동 추정한다는 건 입력 필드를 무시하는 셈.

**보완안**: §2.6에서 자동 추정 로직 삭제. 대신 다음 우선순위:
1. `analysis_config.audience` 명시 → 우선 적용
2. §8 audience 분기 키워드("쉽게/처음" → novice) → 적용
3. 둘 다 없으면 기본값 `intermediate`
4. UI 단계에서 사용자가 명시 선택 권장

### 2.3 [P0] §2.6 audience × §0.5 mode 충돌 해소 규칙 부재

**시나리오**: novice + Full 모드 충돌
- §0.5 Full: "차트 6~8개"
- §2.6 novice: "노출 차트 = line/donut/big_number/underwater 4종"
- 모순: novice가 Full 요청 시 차트 6개 노출해야 하는데 화이트리스트는 4종

**시나리오 2**: novice + §5.2 베타 트리거 발동
- 트리거: "베타 > 자산군 고변동" → 인사이트 자동 삽입
- §2.6 novice: 베타는 노출 KPI 화이트리스트에 없음
- 모순: 인사이트는 생성되는데 KPI는 안 보여줌. 사용자가 인사이트 근거를 검증할 수 없음.

**보완안**: §2.6 표에 "충돌 해소 우선순위" 열 추가:
- 모드 vs audience: audience가 우선 (사용자 친화)
- 트리거 vs audience: 트리거 발동 시 해당 KPI 임시 노출 (검증 가능성)

### 2.4 [P1] 출력 계약 미세 결함 (§6.1)

**A. asset_class enum이 string 타입으로 둔갑**
```typescript
asset_class_thresholds: Record<string, Severity>  // ← string은 너무 느슨
```
자산군은 §1.1에 enum [equity, etf, fund, bond, crypto, commodity]로 정의. 타입을 좁혀야 함:
```typescript
asset_class_thresholds: Partial<Record<AssetClass, Severity>>;
type AssetClass = "equity" | "etf" | "fund" | "bond" | "crypto" | "commodity";
```

**B. KPI.semantic_color 7색 union 비대화**
```typescript
semantic_color: Severity | "positive" | "negative" | "neutral";
// = "info" | "warn" | "critical" | "positive" | "negative" | "neutral"
```
프론트 색상 매핑 로직이 복잡. 의미 분리:
- KPI 색상 = 값 방향성(positive/negative/neutral)
- Insight 색상 = 심각도(info/warn/critical)
- 두 차원을 한 필드에 합치지 말 것

**C. Insight.icon: string 타입 — 자유 문자열**
```typescript
icon: string;  // "⚠️" | "⚡" | "🔻" | "🎯" | "🔗" | "📈"
```
주석에 6개 후보 나열했으나 타입은 자유. literal union으로 제약하면 LLM 환각 방지.

### 2.5 [P1] 거래내역 → 잔고 변환 (§1.2) 갭

**현황**: 변환 규칙 5개 명시. 그러나 다음 에지 케이스 부재:
- `dividend`·`split` "자산별 정책 적용" — 구체 정책 없음
- 매도 후 재매수의 평단가 처리 — 이동평균법 외 LIFO/FIFO 선택권 없음
- 매도 시 `fees` 처리 — 매수 fees는 cost에 포함하지만 매도 fees는?
- 환율 변동 시 cost basis 재평가 정책 없음

**보완안**: §1.2에 "자산 이벤트 처리 정책" 서브섹션 추가 (약 20라인).

### 2.6 [P1] §0.5 시연 1 "3초" 정합성 회수 무리수

**현황**: "Standard 콜드 8초" + "p50 1.5s 캐시 적중 사례를 시연용으로"
**문제**: 시연 환경에서 캐시는 실질적으로 비어있을 가능성 높음. "캐시 적중 사례를 시연용"은 실제로는 사전 워밍업이 필요한 시연. 정직하지 않은 정합성 회수.

**보완안 옵션**:
- A. 시연 1 약속을 명시적으로 깨뜨리고 "5~8초 응답 + 점진적 렌더링(skeleton UI)" 명세
- B. Standard 모드의 콜드 목표 자체를 < 5s로 강화 (LLM 호출 최적화·스트리밍 응답)
- C. Quick 모드를 시연 1 기본값으로 사용 ("핵심 지표만 3초")

### 2.7 [P2] §6.3 캐시 정책 모호함

**현황 갭**:
- 해시 함수 미명시 (SHA-256? xxHash?)
- TTL 미정의 (영구? 7일? 사용자 세션?)
- 인사이트별 의존성 추적 메커니즘 부재 ("임계값 의존 인사이트만 무효" 어떻게 식별)
- 동시성 제어 (cache stampede) 정책 없음

**보완안**: §6.3에 구체 구현 힌트 추가 (Decoupling 원칙 위반 우려가 있어 수준 조정 필요).

### 2.8 [P2] §1.4 입력 어댑터 매핑 사전 부분 커버리지

**현황**: 6개 필드만 매핑 (`asset_id`, `asset_name`, `quantity`, `avg_cost`, `executed_at`, `side`)
**누락**: `asset_class`, `sector`, `base_currency`, `transaction_id`, `position_id`, `fees`, `tax_treatment`
**또한**: "매핑 신뢰도 0.8 미만이면 확인" — 신뢰도 산출 알고리즘 미명시 (Levenshtein? 의미 임베딩?)

### 2.9 [P3] 다국어(i18n) 정책 부재

**현황**: §8 트리거 키워드만 한/영 분리. 출력 텍스트(인사이트, 면책, 차트 라벨) 언어 선택 정책 없음.
